  <action>
Create the directory `.github/workflows/` if it does not exist, then create `.github/workflows/release.yml` with the following exact content:

```yaml
name: release

on:
  push:
    tags:
      - "v*"

permissions:
  contents: write

jobs:
  goreleaser:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Set up Go
        uses: actions/setup-go@v5
        with:
          go-version: stable

      - name: Run GoReleaser
        uses: goreleaser/goreleaser-action@v7
        with:
          distribution: goreleaser
          version: "~> v2"
          args: release --clean
        env:
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

Every field in this file is non-negotiable:

- `on: push: tags: ["v*"]` — triggers ONLY on v-prefixed tags (e.g. v1.0.0, v0.1.0-rc1). Does NOT trigger on branch pushes. This is the correct trigger for goreleaser.
- `permissions: contents: write` — goreleaser must create a GitHub Release and upload assets. Without this, the workflow fails with a 403 "Resource not accessible by integration" error.
- `fetch-depth: 0` — goreleaser reads git history to generate changelog. Shallow clones (GitHub's default fetch-depth: 1) cause goreleaser to fail with "couldn't find any tags before current" or produce an empty changelog.
- `go-version: stable` — actions/setup-go v5 reads the `go` directive from go.mod automatically. No hardcoded version needed; will use 1.24.2 from go.mod.
- `goreleaser-action@v7` — current latest stable action. Do NOT use v4, v5, or v6.
- `version: "~> v2"` — pins to goreleaser v2.x without locking to a specific patch version.
- `args: release --clean` — `--clean` is the v2 flag (replaces deprecated `--rm-dist`). Do NOT use `--rm-dist`.
- `GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}` — auto-provided by GitHub Actions for any repo. No manual secret setup needed for public repos.

Do NOT add any additional steps (linting, testing) to this workflow. The goreleaser action handles everything. If tests are needed in CI, that is a separate workflow file.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Tag push → GitHub Actions trigger | Developer-controlled git tag initiates the build |
| GitHub Actions runner → GitHub Releases API | GITHUB_TOKEN (auto-issued, scoped to repo) uploads release assets |
| goreleaser-action → goreleaser binary | GitHub Action downloads goreleaser binary from goreleaser.com CDN |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-28-04 | Tampering | goreleaser-action download | accept | goreleaser-action@v7 is pinned to a specific major version; goreleaser.com CDN is the official distribution channel |
| T-28-05 | Elevation of Privilege | GITHUB_TOKEN scope | mitigate | `permissions: contents: write` is the minimum required scope; no broader permissions granted (no packages:write, no id-token:write) |
| T-28-06 | Spoofing | Tag-triggered release | accept | Only users with push access to the repo can push tags; GitHub repo access control handles this |
| T-28-07 | Denial of Service | Workflow triggered by non-release tags | accept | `v*` pattern is narrow; accidental triggers produce a failed or empty release that can be deleted; no data loss |
</threat_model>
