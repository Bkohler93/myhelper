## Phase Details

### Phase 2: History & Token Infrastructure
**Goal**: The foundation for conversation is in place — history accumulates and tokens are counted
**Depends on**: Phase 1 (v1.0 baseline)
**Requirements**: HIST-01, HIST-02, HIST-03, CONF-01
**Success Criteria** (what must be TRUE):
  1. Each user turn and model response is stored in an in-memory history structure during a session
  2. After each turn, the total token count of the history is calculated using go-tiktoken
  3. Developer can set the token threshold via an environment variable or CLI flag; the default is 4,100 tokens
  4. The history and token count are accessible to the conversation loop and summarization logic
**Plans**: 3 plans

Plans:
- [x] 02-01-PLAN.md — Extend Config with TokenThreshold, update local config path, add --token-limit flag
- [x] 02-02-PLAN.md — Create internal/history package with Message, History, token counting (TDD)
- [x] 02-03-PLAN.md — Replace StreamPrompt with StreamChat (/api/chat), update all 4 command files

### Phase 3: Conversation Loop
**Goal**: All 4 query commands are interactive — after the first response, the user can ask follow-ups without restarting
**Depends on**: Phase 2
**Requirements**: CONV-01, CONV-02, CONV-03, CONV-04
**Success Criteria** (what must be TRUE):
  1. After receiving the first model response, the user is prompted for a follow-up question without restarting the command
  2. User can type "quit" and the session ends cleanly with no error output
  3. User can press Ctrl+C and the session ends cleanly with no error output
  4. The conversation loop is present in plan, lookup, starter, and pattern — init remains one-shot
**Plans**: 2 plans

Plans:
- [x] 03-01-PLAN.md — Implement runConversationLoop in cmd/helpers.go with SIGINT, quit, empty-reprompt (TDD)
- [x] 03-02-PLAN.md — Wire runConversationLoop into plan, lookup, starter, pattern commands

### Phase 4: Summarization & Re-condensation
**Goal**: History never overflows the model context — the model compresses its own history when the token threshold is hit
**Depends on**: Phase 3
**Requirements**: SUMM-01, SUMM-02, SUMM-03
**Success Criteria** (what must be TRUE):
  1. When the history token count exceeds the threshold, the model is called with a summarization prompt before the next user turn is sent
  2. After summarization, the full conversation history is replaced with the compact summary
  3. When the threshold is hit again in the same session, the prior summary and new turns are condensed together into one new summary
**Plans**: 2 plans

Plans:
- [x] 04-01-PLAN.md — Add history.Replace method (TDD) and ollama.Chat non-streaming function (TDD)
- [x] 04-02-PLAN.md — Wire summarization trigger, summarize(), and re-condensation into runConversationLoop
