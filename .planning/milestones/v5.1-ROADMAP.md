# Milestone v5.1: Configuration Validation & Setup Hardening

**Status:** ✅ SHIPPED 2026-05-10
**Phases:** 31-32
**Total Plans:** 3

## Overview

Removed all hardcoded model/endpoint defaults from `config.Load()` and added startup validation that hard-fails `chat`, `inspect`, and `search` when model or endpoint are not configured — users now get a clear "run myhelper setup" error instead of a cryptic Ollama connection failure. Setup wizard hardened to always write a usable model and endpoint to config before exiting, including a fallback prompt when the user declines the recommended model pull.

## Phases

### Phase 31: Config Loading & Startup Validation

**Goal**: myhelper refuses to run without explicit model and endpoint configuration, and tells the user exactly how to fix it
**Depends on**: Phase 30
**Requirements**: CFG-01, CFG-02, VAL-01, VAL-02, VAL-03, VAL-04, VAL-05
**Plans**: 2 plans

Plans:
- [x] 31-01-PLAN.md — Remove hardcoded defaults from config.Load() and add CFG-01/CFG-02 tests
- [x] 31-02-PLAN.md — Add validateConfig helper and call sites in root.go and inspect.go

**Success Criteria:**
1. Running `myhelper chat` with no config or env produces a clear error message and a "run myhelper setup" hint
2. Running `myhelper inspect` or `myhelper search` with no config produces the same error format as chat
3. Setting `MYHELPER_MODEL` and `MYHELPER_ENDPOINT` env vars allows all three commands to proceed without error
4. Config loading never returns a non-empty default for model or endpoint

### Phase 32: Setup Wizard Hardening

**Goal**: The setup wizard is guaranteed to write a usable model and endpoint to config before it exits
**Depends on**: Phase 31
**Requirements**: WIZ-01, WIZ-02, WIZ-03
**Plans**: 1 plan

Plans:
- [x] 32-01-PLAN.md — Add endpoint prompt (Stage 1.5) and skip-model fallback to wizard

**Success Criteria:**

2. When the user skips the recommended model pull, the wizard prompts them to enter the name of an existing local model before saving and exiting
3. The wizard refuses to write config if the Ollama endpoint field is empty, prompting the user to correct it

## Milestone Summary

**Key Decisions:**
- Hard-fail (not auto-redirect) on missing config — simpler, more predictable than silently launching setup
- Env vars (`MYHELPER_MODEL`, `MYHELPER_ENDPOINT`) count as "set" for validation purposes
- `myhelper config set` subcommand out of scope — setup is the only user-facing config path
- Endpoint prompt moved before reachability check in wizard — remote Ollama users couldn't pass reachability check without first providing their URL
- `pullModel` receives endpoint explicitly — wizard's validated endpoint used for pull, not `ollamaBaseURL`
- `url.Parse` validation for endpoints (not just `HasPrefix`) — bare `http://` passes HasPrefix but has no host

**Issues Resolved:**
- Silent Ollama connection to hardcoded `192.168.0.9:11434` / `qwen2.5-coder:7b` on first run
- Wizard exiting without writing model when user declines pull
- Wizard never writing `endpoint` field to config — remote Ollama users broken after v5.0

**Issues Found During Audit:**
- CFG-01/CFG-02 test isolation gap: test leaked through to real `~/.config/myhelper/config.json` — fixed with `t.Setenv("HOME", t.TempDir())` (commit b0a5e34)
- CR-01 (Phase 32): pullModel used package-level `ollamaBaseURL` instead of user's endpoint — fixed
- CR-02 (Phase 32): reachability check fired before endpoint prompt — reordered

**Technical Debt Incurred:**
- `validateConfig` combined-error branch is dead code (endpoint-only branch fires first)
- Wizard endpoint-probe tests take ~5s each (HTTP timeout on custom endpoint)
- `search_test.go` has same HOME isolation gap as Phase 31 config tests — outside v5.1 scope
