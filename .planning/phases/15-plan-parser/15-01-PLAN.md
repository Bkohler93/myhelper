Create `internal/planner/planner.go` with package `planner`. Use only stdlib imports: `bufio`, `fmt`, `os`, `strconv`, `strings`.

Implement the parsing in two phases:

**Phase 1 — Split frontmatter from body:**
Read the full file content as a string. Find the first `---` line and the second `---` line. The frontmatter is the text between them. The body is everything after the second `---` line. If fewer than two `---` delimiter lines exist, return `fmt.Errorf("parse plan: missing frontmatter delimiters")`.

**Phase 2 — Parse frontmatter with bufio scanner:**
Use `bufio.NewScanner(strings.NewReader(frontmatter))` and loop with `sc.Scan()`. Track an `inFilesModified bool` flag. For each line:
- `trimmed := strings.TrimSpace(line)`
- If `strings.HasPrefix(trimmed, "phase:")` → `p.Phase = strings.TrimSpace(strings.TrimPrefix(trimmed, "phase:"))`; set `inFilesModified = false`
- If `strings.HasPrefix(trimmed, "plan:")` → parse int; set `inFilesModified = false`
- If `strings.HasPrefix(trimmed, "wave:")` → parse int; set `inFilesModified = false`
- If `strings.HasPrefix(trimmed, "autonomous:")` → `p.Autonomous = strings.TrimSpace(strings.TrimPrefix(trimmed, "autonomous:")) == "true"`; set `inFilesModified = false`
- If `strings.HasPrefix(trimmed, "files_modified:")` → set `inFilesModified = true`; continue
- If `inFilesModified && strings.HasPrefix(line, "  - ")` → append `strings.TrimSpace(strings.TrimPrefix(line, "  - "))` to `p.FilesModified`
- If `inFilesModified && !strings.HasPrefix(line, "  - ") && trimmed != "" && !strings.HasPrefix(trimmed, "#")` → set `inFilesModified = false` (new key started)

**Phase 3 — Extract XML task blocks from body:**
Use `strings.Index` / `strings.Index` scanning approach: find `<task` and `</task>` pairs in order. For each matched block (from `<task` to `</task>` inclusive), call a helper `extractElement(block, tag string) string` that returns inner text of `<tag>...</tag>` (handles multi-line: find `<tag>` then `</tag>`, capture everything between, trim whitespace).

Build each `Task`:
```go
t := Task{
    Name:     extractElement(block, "name"),
    Files:    extractElement(block, "files"),
    Behavior: extractElement(block, "behavior"),
    Action:   extractElement(block, "action"),
}
```

Validate each task before appending. Return the first validation error encountered (task index is 1-based in the error message).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| filesystem → ParsePlan | File content from `.planning/phases/` crosses from disk into the parser; content is author-controlled but could be malformed or crafted |
| caller → ParsePlan path arg | Path string from Phase 18 execute command; could traverse outside expected directory |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-15-01 | Tampering | ParsePlan path argument | accept | Path is supplied by `FindActivePlan` (same package) or the `execute` command's own directory scan — not from user-supplied free text. No path sanitization needed at the parser level; the caller controls the path. |
| T-15-02 | Information Disclosure | parse error messages | mitigate | Error messages include task index and missing field name but NOT the full file path or file contents. Use fmt.Errorf("parse plan: task %d missing <name>", i+1) — no path interpolation in inner error messages. The outer "read plan: %w" wraps the os.ReadFile error which may include the path; this is acceptable (same pattern as config.go). |
| T-15-03 | Denial of Service | billion-laughs / deeply nested XML | accept | Parser does not use encoding/xml — it uses strings.Index for tag boundaries. Deeply nested XML has no recursive descent risk. A maliciously large `<action>` block is bounded by the file size; os.ReadFile on a >100MB file would exhaust memory, but PLAN.md files are always <100KB in practice. No explicit size cap added (ASVS L1 does not require it for local-only tooling with trusted input). |
| T-15-04 | Denial of Service | resource exhaustion from large files | accept | File is read once with os.ReadFile into a single string; no streaming parse loops that could be forced into O(n²) behavior. extractElement uses strings.Index which is O(n) per call. Acceptable for local dev tooling. |
| T-15-05 | Information Disclosure | path traversal in fixture test | mitigate | Test fixture path is a hardcoded relative path (`../../.planning/phases/14-.../14-01-PLAN.md`) — not derived from user input. No traversal risk. |
</threat_model>
